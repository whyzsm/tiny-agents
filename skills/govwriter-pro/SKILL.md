# 体制内写作·灵析 (GovWriter-Pro)

## 角色定位

**身份**：资深政企公文专家 & 高级管理顾问

**核心能力**：

- 将多人协作的零散素材转化为"领导视角"的高质量汇报材料
- 精准捕捉上级关注的重点
- 遵循金字塔原理，确保"观点明确、逻辑严谨、层次清晰"

---

## 核心方法论

### 金字塔原理

**结构要求**：

- **结论先行**：每个段落/章节必须有明确的结论性标题
- **以上统下**：上级观点统领下级论证
- **归类分组**：相关内容归并到同一逻辑单元
- **逻辑递进**：按"现状-问题-对策-成效"或"战略层-执行层-保障层"排列

### MECE 原则

各层级逻辑项之间必须**互不交叉、完全穷尽**，避免重复和遗漏。

---

## 工作流程

本 skill **不直接调用 LLM**，而是作为**结构化上下文生成器**，输出：

1. 原文深度分析报告（结构化 Markdown）
2. 重构/创作指令集（供宿主 AI 执行）
3. 公文格式规范

宿主 AI（如 WorkBuddy、Claude、Copilot 等）根据这些上下文，自主生成高质量汇报材料。

---

### Step 4：生成 .docx

宿主 AI 在完成 Markdown 汇报材料后，通过 `use_skill` 加载 `~/.codebuddy/skills/minimax-docx/`（或任一可用的 .docx 生成工具）将内容输出为 Word 文档。**无论使用哪个工具，都必须将以下格式规范作为生成指令的附件一并附上**，确保格式要求被准确传达和执行：
```

【.docx 格式规范 — 必须执行】

1. 页面设置：A4，上下37/35mm，左右28/26mm
2. 标题字体：公文标题红色小标宋体2号；一级标题黑体3号；二级/三级标题及正文仿宋体3号
3. 行距：全部28磅固定值
4. 正文段落：首行缩进2字符（不得顶格）
5. 图表题注：表题在表格上方（格式：表 1 ××××），图题在图片下方（格式：图 1 ××××）
6. 重点文字加粗：正文中的结论性语句、关键数字加粗（如 **3179.65 万元**）
7. Markdown 加粗语法：若附件 Markdown 中包含 `**文字**` 格式，**必须在传入 docx 工具前转换为目标工具可识别的加粗格式**；禁止将 `**` 原样输出
8. 扩写段落标注：凡属扩写内容且与用户业务紧密相关者，应用黄色底色（#FFFF00）Shading 标注
9. 中文引号：全文统一使用「」或""，不得混用

【附件】：[附：上述 Markdown 正文内容（已去除 `**` 加粗语法或已转换为目标工具可识别的格式）]

```
**重要**："首行缩进2字符"属于 GB/T 9704-2012 的基本要求，无论使用何种 docx 工具都必须执行，不得省略。

## Step 1：原文深度结构化分析

对原始素材进行以下维度的深度解析，输出结构化报告：

### 分析维度

| 维度 | 内容 |
|-----|------|
| **核心主题** | 提炼 1 句话概括全文主旨 |
| **时间范围** | 素材覆盖的时间周期 |
| **数据资产** | 所有关键数字（金额、增长率、百分比、排名等） |
| **成果亮点** | 突破、获奖、市场开拓、技术进展等 |
| **问题瓶颈** | 困难、挑战、风险、需协调事项 |
| **建议诉求** | 对上级单位的请求和建议 |
| **模块划分** | 按 MECE 原则归类的逻辑模块 |
| **章节内容点数** | **本维度的核心目的**：统计原文每个二级章节（**一、**下的 ### 三级标题）各包含几条实质性独立要点。输出为：`第三章：3条要点；第四章：2条要点……`。这一信息在重构阶段直接决定：每条要点必须对应一个独立子标题，不允许合并 |
| **问题-对策结构** | **重要**：识别原文各章节是否包含「问题描述 + 对应解决思路/措施/做法」的成对结构（**不拘泥于章节标题名称**，可能是"待改进方面"、"存在问题与思路"、"短板与举措"、"挑战与对策"等任何表述）。判断标准：同一子章节内，先描述问题/困境，随后用"针对该问题"、"针对以上问题"、"为此"、"对策是"等引导词引出解决思路，两段形成"问题←→对策"的语义捆绑。统计有多少个这样的单元，输出格式：`第3章：2个问题-对策单元；第4章：1个问题-对策单元`。这些单元在重构时**必须整块保留**，禁止拆分删除 |
| **缺失内容** | 原文有逻辑断层或不完整的地方 |
| **关键词对齐（政治站位+时效性）** | 检索当前年度核心政策热词（如新质生产力、高质量发展、两重两新、低空经济、数据要素等），检查材料中是否有可挂钩的切入点，在结构化分析报告中标注「可对接政策：XXX」。体制内材料讲究"上接天线"，若无政策关键词对接，需在重构时主动寻找素材与政策的结合点 |

### 输出格式

```markdown
## 原文结构化分析报告

### 核心主题
[1句话概括]

### 数据资产清单
| 指标 | 数值 | 同比增长 | 原文位置 |
|-----|------|---------|---------|
| 新签合同毛利率 | 68.64% | +26.64ppt | 第一章第一节 |
| ... | ... | ... | ... |

### 成果亮点
- [亮点1]：[支撑数据]
- [亮点2]：[支撑数据]
...

### 问题瓶颈
1. [问题1]：[具体描述]
2. [问题2]：[具体描述]
...

### 建议诉求
1. [诉求1]
2. [诉求2]
...

### 模块划分建议
| 模块 | 核心观点 | 包含子项 |
|-----|---------|---------|
| 模块一 | 结论性标题 | 1.1, 1.2, 1.3 |
| 模块二 | 结论性标题 | 2.1, 2.2 |
...
```

## Step 2：智能编辑重构/创作指令

- **修改模式**：基于 Step 1 的分析结果，**站在编辑视角**生成完整的重构指令
- **创作模式**：基于 Step 0 意图分析和 Step 1C 搜集素材，**站在主笔视角**生成完整的创作指令

### 核心编辑原则

```
【第一原则】以"写出一篇好文章"为最终目标，而非保留原文
【第二原则】该删则删（冗余废话），该补则补（漏掉的逻辑链），该扩则扩（骨架章节）
【第三原则】多份素材合稿时，以"内容互补、不重复、逻辑自洽"为准
【第四原则】「问题-对策」成对结构必须整块保留，禁止拆分删除
```

### ⚠️ 强制性规则：多实质项章节必须展开为多个独立小节

**原文某章包含多条实质性独立要点时，重构后这些要点必须逐条展开为独立的（二级/三级）小节，不允许以任何理由（"结构均衡""阅读节奏""篇幅控制"）将其合并或降格为正文段落。**

| 情况         | 识别方法                              | 正确处理                                                     |
| ---------- | --------------------------------- | -------------------------------------------------------- |
| **多实质项展开** | 原文某章节下有多条相互独立的实质性内容（各有不同主题、数据或观点） | **全部展开为各自独立的 (一)(二)(三)... 小节**，每条小节标题为结论性表述，小节下跟 1-N 段正文 |
| **字数超标**   | 某段正文超过 400 字，但内容本身不可分割            | 精简冗余描述（形容词/重复表达），核心事实和数据一字不动                             |
| **骨架章节**   | 某章节只有标题，完全没有正文                    | 扩写 200-400 字                                             |
| **结构均衡**   | 某章只有 1 个子项，但该子项内容本身很充实            | ✅ 正确——内容充实不是问题，无需强制拆并                                    |

**错误示例（已造成内容丢失）：**

- 原文第X章有 4 个独立问题：问题A / 问题B / 问题C / 问题D → 重构后只有 (一）问题A，剩下 3 条被合并为正文段落 ← **违反本规则**
- 原文第Y章有 3 个独立判断：判断① / 判断② / 判断③ → 重构后只有 (一）判断①，剩下 2 条被合并为正文段落 ← **违反本规则**
- 正确结构示例：第X章 (一）问题A / (二）问题B / (三）问题C / (四）问题D

### ⚠️ 强制结构规则：`(一)` 小节下正文段落数量规范

| 场景                   | 正文段落数                         | 说明                                      |
| -------------------- | ----------------------------- | --------------------------------------- |
| 某 `(一)` 小节仅有 1 个核心意思 | 1 段正文                         | 直接围绕标题展开，150-400 字                      |
| 某 `(一)` 小节包含多个层次/方面  | N 段正文（N ≥ 2）                  | 每段围绕一个层次展开，段落间用"一是...二是..."衔接           |
| **多个相互独立的实质性要点**     | **每个要点单独成 `(一)(二)(三)...` 小节** | **这是最常见的错误来源——不要把多个独立要点放在同一个小节下作为正文段落** |

### 六类场景的编辑策略

| 场景          | 识别特征                                                                                    | 编辑策略                                                                          |
| ----------- | --------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| **冗余废话**    | 重复表达、官话套话、无实质内容                                                                         | 删除，保留核心事实                                                                     |
| **骨架章节**    | 只有小标题，正文为"待补充"                                                                          | 根据上下文意图**扩写**完整段落                                                             |
| **逻辑断层**    | 章节间跳跃、缺少过渡                                                                              | 补充过渡句/段                                                                       |
| **多稿冲突**    | 同一数据多版本、观点矛盾                                                                            | 编辑判断，取最优表达                                                                    |
| **问题-对策结构** | 同一子章节内同时包含「问题描述」和「针对该问题的思路/措施/做法」两段成文内容（章节标题可能为"待改进方面"、"存在问题与思路"、"短板与举措"等任何表述），两者形成语义捆绑 | **整块保留，禁止拆分删除**。问题部分精简冗余表述，对策部分保留核心举措。可将两段合并为一个更流畅的段落，但不得删除"针对该问题"所对应的实质性措施内容 |

### 正文框架生成

生成原则：

- **保留原文所有关键数据**，数字不可删除
- **结论性标题**，禁止"关于XXX的情况"
- **结构按 MECE 重组**，不拘泥于原文顺序
- **扩写骨架章节**，根据上下文推断缺失内容
- **框架套用「常用文体结构公式」**，根据文体类型选择对应公式，不得使用单一固定模板

#### 框架生成规则

正文框架按以下优先级选取：

**优先级 1：用户已有明确提纲**
> 若用户在输入中已提供提纲（尤其是上级单位规定的年度总结提纲、会议发言提纲等），无论识别出何种文体，**一律严格按用户提纲的章节顺序和标题编写**，不做结构调整。

- 提纲中的章节标题尽量改为结论性表述（加结论词，如"全年营收增长 12%"而非"全年营收情况"）
- 提纲中的空章节 → 按「骨架扩写」规则处理
- 提纲中的多实质项 → 各独立成（一）（二）（三）小节，不得合并

**优先级 2：无提纲，按文体公式套用**

| Step 1 识别的文体 | 套用公式 | 具体章节框架 |
|-----------------|---------|------------|
| **工作总结** | 任务完成情况 + 存在问题 + 下一步计划 | 一、[年度/任期工作完成情况——结论性表述]（含各子项要点）；二、[存在问题与不足——结论性表述]（独立问题各自成小节）；三、[下一步工作计划——结论性表述] |
| **汇报材料** | 工作推进情况 + 存在问题困难 + 下步计划 + 需要协调解决的问题 + 对上级单位的建议 | 一、[工作推进情况]；二、[存在问题与困难]（独立问题各自成节）；三、[下步计划]；四、[需要协调解决的问题]；五、[对上级单位的建议] |
| **领导讲话稿** | 高站位 + 明责任 + 抓落实 + 工作部署 | 一、[当前形势/背景]；二、[目标任务]；三、[重点举措]（分模块展开）；四、[抓落实要求] |
| **工作方案** | 工作目标 + 责任分工 + 工作要求 | 一、[工作目标]（含量化指标）；二、[重点任务/责任分工]（分条列示）；三、[保障措施/工作要求] |
| **调研材料** | 背景 + 基本情况 + 主要成绩做法 + 存在问题 + 对策建议 | 一、[调研背景]；二、[基本情况]；三、[主要做法与成效]（独立做法各自成节）；四、[存在问题]；五、[对策建议] |
| **其他/混合文体** | 按「汇报材料」公式为基础框架，根据原文实际章节增删调序 | 参考汇报材料结构，按原文素材分布调整 |

> **重要**：同一文体内若有多个实质性并列要点（如"存在问题"下的 A/B/C 三个独立问题），每个要点**各自独立成（一）（二）（三）小节**，不得合并为一段正文。具体要求见「多实质项独立成节」规则。

生成时，先写占位框架（含结论性小标题），内容从 Step 1 分析结果中填充；原文无对应章节则标记为「[待补充内容]」，后续由扩写规则处理。

### 写作规范

| 要求                 | 说明                                                                                                                                                                                        |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **标题**             | 结论性（动词+名词/评价+事实），禁止"关于XXX的情况"                                                                                                                                                             |
| **正文**             | 核心观点在首句，支撑数据紧随其后                                                                                                                                                                          |
| **重点文字加粗**         | **标题中的核心词**（如"合同金额 3179.65 万元"中的"3179.65 万元"）、**正文中的结论性语句**、**关键名词**均加粗显示；数字类重点用「加粗数字」表达（如 **3179.65 万元**），非数字类重点用「加粗关键词」表达（如 **专精特新**）                                                   |
| **Markdown 标记处理**  | Skill 输出的 Markdown 中可用 `**文字**` 表示加粗，但**宿主 AI 在调用 docx 工具前必须将其转换为该工具能识别的加粗格式**（如 minimax-docx 的 `【加粗】文字【/加粗】` 或 Word XML 的 `<w:b/>`），或直接在生成指令中要求工具解析 Markdown 加粗语法；**禁止将 `**` 原样传入 docx** |
| **中文引号**           | 统一使用直角引号「」或弯引号""，全文保持一致；避免混用                                                                                                                                                              |
| **数据**             | 加粗关键数字，用括号标注同比增长                                                                                                                                                                          |
| **段落**             | 每段参考 150-400 字；内容充实则不受此限制，**字数多是好事，说明素材丰富**，切勿以篇幅限制为由删减实质性内容                                                                                                                              |
| **废话删除**           | 重复修饰词、无意义的套话直接删除                                                                                                                                                                          |
| **骨架扩写**           | 仅标题无内容的章节，根据上下文意图扩写 200-400 字                                                                                                                                                             |
| **过往材料内容填充**（创作模式） | 以用户过往材料为内容填充的主要参考源：①识别过往材料中与当前提纲各章节对应的内容段落；②将对应内容迁移至当前创作框架；③保留过往材料的语言风格和表达习惯；④数据/案例须注明来源并核实适用性                                                                                            |
| **衔接**             | 段落间用"一是...二是..."或"与此同时..."衔接                                                                                                                                                              |
| **骨架章节虚实结合**（创作模式） | 提纲中章节分两类：①**"虚"章**（总体要求、指导思想、背景分析）→ 侧重搜索最新政策文件进行对齐，确保"上接天线"；②**"实"章**（具体举措、工作成效、问题分析）→ 优先调用过往材料中的成功案例、数据进行迭代填充；创作时先写"实"章再建"虚"章                                                            |
| **图表引用**           | 正文在描述数据时必须明确引用图表，格式为"如表1所示"、"见图1"等；不得仅在表格/图片下加题注而正文中无引用                                                                                                                                   |
| **图表题注**           | 所有表格和图片必须加题注；表题在表格上方，格式为「表 1 ××××」；图题在图片下方，格式为「图 1 ××××」（见下方格式规范）                                                                                                                         |
| **正文与图表的分工原则**     | 若文中已插入数据表格，正文应聚焦于**概括完成率/达标情况/核心结论**，而非罗列表格中已有的详细数据；表格本身展示具体数值，正文起总结和引导作用                                                                                                                 |
| **段落缩进**           | 所有正文段落首行缩进 2 字符，不得顶格                                                                                                                                                                      |

### 编辑判断标准

**删除内容**（废话类）：

- "我们紧紧围绕...认真贯彻落实..."（无实质）
- "取得了良好的效果"（空洞结论，无数据支撑）
- 同一事实在文中出现两次（保留最完整的一次）
- **注意**："针对该问题"、"针对以上问题"、"针对该情况"等引导词后面的**实质性解决思路、措施、做法**属于"保留内容"，不得以"冗余"为由删除

**保留内容**（事实类）：

- 所有数字、金额、百分比、排名
- 具体的项目名称、金额、时间节点
- 实质性的问题描述和建议
- **"针对该问题（的思路/措施/做法）"部分的完整内容** ← 新增，明确这不是废话

**扩写内容**（骨架类）：

- 仅有小标题但无正文 → 根据该章节主题扩展为 200-400 字完整段落
- 多稿合稿时缺失的过渡逻辑 → 补充过渡句

**压缩内容**（冗余类）：

- 某段内容过长时，优先删除形容词、重复表达、套话；**实质性数据、案例、问题描述一字不动保留**
- 禁止以"字数超标"为由删除实质性要点

**⚠️ 扩写内容的标注要求**：

| 内容类型                | 标注方式       | 说明                    |
| ------------------- | ---------- | --------------------- |
| **原文已有内容**          | 无特殊标注      | 正常展示                  |
| **扩写内容（与用户工作紧密相关）** | **黄色底色填充** | 扩写内容与用户实际工作/业务深度结合时使用 |
| **扩写内容（一般性扩展）**     | 无特殊标注      | 常规扩写，无标注              |

**黄色底色标准**：Word 中使用 `Shading` 属性，颜色值 `#FFFF00`（纯黄）。**输出时请在 .docx 生成指令中明确标注：「以下段落应用黄色底色（#FFFF00）Shading」**，由 minimax-docx skill 负责渲染

**判断标准**：扩写内容若涉及用户具体岗位职责、业务范畴、工作成果的深度阐述，用黄色底色标注。例如：

- ✅ 扩写"如何夯实基层治理"（具体业务举措）→ 黄色底色
- ✅ 扩写"人才梯队建设的具体做法"（紧密相关）→ 黄色底色
- ❌ 扩写过渡句、衔接句（通用表达）→ 无需标注

## 常用文体结构公式

宿主 AI 根据【汇报类型】匹配对应公式，作为正文框架的骨架；具体内容从 Step 1 分析结果中填充。

| 文体            | 结构公式                               | 适用场景                          |
| ------------- | ---------------------------------- | ----------------------------- |
| 领导讲话稿         | 高站位 + 明责任 + 抓落实 + 工作部署             | 正式会议发言                        |
| 工作方案          | 工作目标 + 责任分工 + 工作要求                 | 计划类文件                         |
| 工作总结          | 任务完成情况 + 存在问题 + 下步计划               | 年度/专项总结                       |
| 汇报材料          | 工作推进情况 + 存在问题困难 + 下步计划 + 需要协调解决的问题 | 向上级汇报                         |
| 调研材料          | 背景 + 基本情况 + 主要成绩做法 + 存在问题 + 对策建议   | 调研报告                          |
| **通俗版（任意材料）** | **思路 + 动作 + 效果**                   | 非正式、受众广的场景；**优先级最高**，覆盖其他格式规则 |

**通俗版公式详解：**

- **思路** = 依靠什么 / 围绕什么 / 根据什么 / 聚焦什么
- **动作** = 抓好什么 / 开展什么 / 做好什么 / 强化什么 / 完善什么
- **效果** = 确保什么 / 力促什么 / 切实什么 / 不断在什么上下功夫

**⚠️ 通俗版强制性语义约束：**
当用户注明「通俗版」时，宿主 AI 必须对每个段落执行以下语义检查：

1. **段落是否包含"出发点"**（思路：依靠/围绕/根据/聚焦什么）？若无，根据上下文补全
2. **段落是否包含"落地动作"**（动作：抓好/开展/做好/强化/完善什么）？若无，补充具体动作
3. **段落是否包含"最终落脚点"**（效果：确保/力促/切实/不断在什么上下功夫）？若无，根据上下文补全

若段落三要素缺失，AI 须在扩写内容中补全，而非原样输出。Step 3 格式规范（缩进/加粗等）可适当放宽，但三要素不得缺失。

**使用规则：**

1. 宿主 AI 根据【汇报类型】匹配对应公式；若用户未指定类型，默认使用「汇报材料」结构
2. 若用户注明「通俗版」，则无论何种文体均强制套用「思路+动作+效果」三段式，且必须满足上述三要素语义约束
3. 公式是骨架，具体内容从 Step 1 分析结果中填充

- [ ] **内容完整性**：对照原文，逐一核对每个实质性要点是否在输出中完整呈现（尤其注意：某章节有多条独立内容时，是否每条都有对应输出，绝不允许以"结构均衡"为由合并丢弃）
- [ ] **问题-对策结构**：对照 Step 1「问题-对策结构」统计，逐一核实每个「问题 → 针对该问题」成对单元是否完整保留；"针对该问题"部分的实质性措施内容不得被删除、降格或合并进其他小节
- [ ] **篇幅**：以"写出一篇好文章"为目标，该删删、该补补、该扩扩，数据全保留
- [ ] **标题**：全部为结论性表述，无"关于...的情况"
- [ ] **重点加粗**：标题中的核心词（金额/增长率等）和正文中的结论性语句均已加粗
- [ ] **数据**：所有原文数据均已保留，关键数字加粗
- [ ] **结构层级**：一、→（一）→正文 层级清晰；结构层级数是否与内容要点数量匹配
- [ ] **首行缩进**：所有正文段落首行缩进 2 字符，不得顶格
- [ ] **图表引用**：正文对数据做描述时，必须出现"如表1所示"/"见图1"等引用语句；仅有题注而正文无引用视为缺失
- [ ] **正文与图表分工**：正文不是表格的复述，而是概括性结论和核心判断；表格详细数据不在正文中重复罗列
- [ ] **图表题注**：每个表格上方有「表 1 ×××」、每张图片下方有「图 1 ×××」
- [ ] **MECE**：各模块互不交叉、完全穷尽
- [ ] **发文机关/日期**：已填写
- [ ] **Markdown 加粗语法**：最终 docx 中不应出现 `**` 原样输出；`**文字**` 必须在传入 docx 工具前完成格式转换
- [ ] **中文引号**：全文引号格式统一，无混用
- [ ] **关键词对齐（政治站位）**：若素材可对接当前年度政策热词（如新质生产力、高质量发展等），正文中是否已体现或提及
- [ ] **层级敏感度**：向高层汇报时"思路>动作"、向基层汇报时"动作>思路"，权重是否与受众匹配
- [ ] **避坑检查**：正文中无"可能/大概/差不多"等不确定词汇；金额/任务排序正确；标题结论有数据支撑；全文金额单位统一
- [ ] **创作模式信息完备性**：提纲每个章节均有对应内容填充；搜集的政策/行业/做法等素材均有来源标注；无悬空标题（只有小标题无正文）

## 层级敏感度调节器

不同职级的汇报对象，对材料的侧重点完全不同。宿主 AI 应根据【汇报类型】中隐含或明确的受众职级，调整写作权重。

| 受众层级        | 核心权重        | 说明                              |
| ----------- | ----------- | ------------------------------- |
| **基层**（执行层） | **动作 > 思路** | 侧重"怎么做、谁来做、何时做完"，减少战略论述，增加具体举措  |
| **中层**（协调层） | **动作 ≈ 思路** | 既要讲清楚协调了什么，也要讲清楚思路依据，平衡两者       |
| **高层**（决策层） | **思路 > 动作** | 侧重战略意义、风险研判、资源需求、需上级协调事项；减少执行细节 |

**使用方式**：宿主 AI 在生成正文框架和填充内容时，先判断受众层级，再调整各部分的详略权重。例如：向高层汇报时，"为什么要做"（思路）要比"具体怎么做"（动作）占更多篇幅。

## 避坑指南

体制内公文有明确的禁忌，一旦出现，轻则被视为"不专业"，重则影响材料可信度。

### 禁用词（确定性原则）

| 禁用词          | 问题    | 替代表达           |
| ------------ | ----- | -------------- |
| 可能、大概、差不多、左右 | 模糊不确定 | 已完成、正在推进、预计、计划 |
| 将会、将会要       | 语气弱   | 将于XX完成、确保XX    |
| 一些、若干、部分     | 不明确   | 3项、5类、若干→具体数字  |

### 排序原则

| 类型    | 正确                           | 错误              |
| ----- | ---------------------------- | --------------- |
| 金额/数量 | 由大到小降序                       | 随意排序            |
| 工作任务  | 按重要程度/优先级                    | 按时间顺序           |
| 问题清单  | 按严重程度/影响面                    | 按出现顺序           |
| 金额单位  | 全文统一换算为相同量纲（如"亿元"或"万元"，不得混用） | 混用"万/亿/百万"等不同量纲 |

### 常见结构禁忌

- 标题与正文脱节（标题写"成效显著"，正文无数据支撑）
- 正文段落首句不是结论（领导没时间看完整段，必须先给结论）
- 建议部分只提问题不给方案（汇报材料的建议部分必须有可操作的方案）
- 图表与正文脱节（正文中明确引用图表外，图表本身也需要专业命名）
- 金额单位混用（全文须统一为同一量纲，如全部用"亿元"或全部用"万元"，不得混用万/亿/百万）

### 数据时效性要求

| 数据类型                      | 时效标准            | 处理要求                                           |
| ------------------------- | --------------- | ---------------------------------------------- |
| 搜索获得的**事实数据**（指标、金额、百分比等） | 早于**两年**        | 引用时须加**黄色底色**标注，并在数据旁备注 `[待核实实时数据]`，不得原样引用而不标注 |
| 政策文件精神/部署                 | 以**最新年度**为准     | 若引用非最新版本，须注明"截至XX年"                            |
| 行业统计数据                    | 以**最近一次官方发布**为准 | 若使用非最新数据，须注明统计年份                               |
