# 临床诊断推理框架（Diagnostic Reasoning Framework）

## 1. 输入解析规范

### 1.1 非结构化输入（病历文本）

从自由文本中提取以下要素：

| 要素类别 | 提取目标 | 示例 |
|---------|---------|------|
| 主诉 | 主要症状 + 持续时间 | "反复口渴多饮3月" |
| 现病史 | 发病经过、诱因、伴随症状、演变过程 | 起病方式、加重/缓解因素 |
| 既往史 | 基础疾病、手术史、用药史 | 高血压10年、阑尾切除术后 |
| 个人史 | 吸烟/饮酒/职业暴露/旅居史 | 吸烟20年×1包/日 |
| 家族史 | 遗传性/家族聚集性疾病 | 父亲糖尿病 |
| 体格检查 | 阳性体征 + 关键阴性体征 | T 38.5°C、Murphy征阳性 |
| 辅助检查 | 实验室、影像、病理等结果 | HbA1c 8.2%、CT示占位 |

### 1.2 结构化输入（JSON 格式）

接受如下结构：

```json
{
  "chief_complaint": "主诉文本",
  "present_illness": "现病史文本",
  "past_history": "既往史",
  "personal_history": "个人史",
  "family_history": "家族史",
  "physical_exam": {
    "vitals": {},
    "findings": []
  },
  "lab_results": [
    { "item": "指标名", "value": "数值", "unit": "单位", "reference": "参考范围", "abnormal": true }
  ],
  "imaging": [
    { "modality": "检查类型", "findings": "结果描述" }
  ],
  "current_diagnosis": "当前诊断（如有）"
}
```

## 2. 诊断推理方法论

### 2.1 假设-演绎法（Hypothetico-Deductive）

1. 根据主诉和核心症状生成初始假设列表（3-7个）
2. 按概率排序：常见病优先（"When you hear hoofbeats, think horses, not zebras"）
3. 逐一检验每个假设与已有证据的匹配度
4. 保留匹配度高的，淘汰矛盾的

### 2.2 模式识别（Pattern Recognition）

- 识别经典症候群模式（如 Cushing 三联征、Charcot 三联征）
- 匹配"一元论"解释——优先用单一疾病解释全部表现
- 当一元论无法解释时，考虑共病或并发症

### 2.3 安全网思维（Safety Netting）

- 始终考虑"不能漏诊"的危重疾病（red flags）
- 即使概率低，若后果严重则必须列入鉴别清单
- 关注：急性冠脉综合征、肺栓塞、主动脉夹层、脑卒中、脓毒症、恶性肿瘤等

## 3. 四类决策输出规范

### 3.1 疑似诊断（无当前诊断 + 最高可能性为"高"时触发）

**触发条件**：
1. 输入中无明确的当前诊断（`current_diagnosis` 为空或未提供）
2. Phase 2 推理后，至少有 1 条疑似诊断的可能性评估为"高"

**若最高可能性仅为"中"或"低"**：禁止输出疑似诊断报告，强制转入 3.4 缺失信息识别。

输出结构：
- 仅输出可能性为"高"的疑似诊断（不输出"中"/"低"条目）
- 每条包含：
  - **诊断名称**：标准疾病名称
  - **诊断依据**：从输入信息中逐条列出支持该诊断的阳性发现
  - **可能性评估**：高
  - **需进一步确认的关键检查**

### 3.2 鉴别诊断（有当前诊断时触发）

**触发条件**：输入中已有当前诊断

输出结构：
- 列出 3-5 个需鉴别的疾病
- 每条包含：
  - **鉴别诊断名称**
  - **与当前诊断的相似点**：哪些表现可被两种诊断共同解释
  - **鉴别要点**：区分两者的关键差异（症状/体征/检查）
  - **排除依据**：当前信息中哪些发现不支持该鉴别诊断
  - **无法排除时的建议检查**

### 3.3 误诊/漏诊风险评估（有当前诊断时触发）

**触发条件**：输入中已有当前诊断

输出结构：

#### 误诊风险
- **可能误诊为**：当前诊断可能实际是哪种疾病
- **误诊原因**：哪些表现容易导致混淆
- **鉴别依据**：支持重新考虑的关键线索

#### 漏诊风险
- **可能漏诊的共病/并发症**
- **漏诊原因**：哪些症状被当前诊断"掩盖"
- **排查依据**：提示需进一步关注的线索

### 3.4 缺失信息识别（信息不足或置信度不足时触发）

**触发条件**（满足任一即触发）：
1. 现有信息明显不完整（如仅有主诉无辅助检查）
2. Phase 2 推理后，所有疑似诊断的最高可能性仅为"中"或"低"——置信度不足等价于信息不充足

输出结构：
- **是否缺失诊断依据**：是/否
- 缺失清单（逐条列出）：
  - **缺失类型**：症状 / 实验室指标 / 体征 / 影像学病灶 / 解剖部位
  - **缺失内容**：具体缺什么（如"缺少甲状腺功能检查"）
  - **理由**：为什么需要这项信息
  - **依据路径**：从哪条推理链路得出需要该信息
    - 格式：`症状A + 体征B → 疑似疾病C → 确诊需指标D`

## 4. 三层验证门控

所有诊断推理结果在输出前必须通过三层验证。详细规则见 `validation_rules.md`。

### 4.1 验证架构

| 层级 | 名称 | 职责 | 判断方式 |
|------|------|------|---------|
| 第一层 | 规则验证 | 硬性规则校验（R01-R08） | 规则命中即判定，自动修正 |
| 第二层 | 模型交叉验证 | 临床合理性审视（V01-V05） | 规则不存在/无法判断时，模型兜底 |
| 第三层 | 输出完整性终检 | 结构/安全/溯源终检（F01-F07） | 全部通过才放行 |

### 4.2 核心原则

- 规则能判的，规则先判——确定性高、速度快
- 规则不存在或规则无法判断的，模型兜底——覆盖临床合理性等软判断
- 结果不正确不输出——宁可回退修正，不带错放行
- 最多回退 2 次，超限则带不确定性警告输出

## 5. 安全声明

- 所有输出仅供辅助参考，不构成最终医学诊断
- 最终诊断必须由执业医师结合临床实际做出
- 危急值/急症提示应在输出中醒目标注
