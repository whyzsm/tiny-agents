# 三层验证门控规则详解（Validation Rules Reference）

## 概述

所有诊断推理结果在输出前必须通过三层验证。三层设计的核心思路：

- **第一层（规则验证）**：确定性高的错误由硬规则捕获，命中即判定，自动修正
- **第二层（模型交叉验证）**：规则无法覆盖或无法判断的问题，由模型以"质疑者"角色兜底审视
- **第三层（输出完整性终检）**：最终放行前的结构性/安全性检查

```
判断优先级：规则能判 → 规则判定
            规则不存在 → 模型兜底
            规则无法判断 → 模型兜底
```

---

## 第一层：规则验证（Rule-based Validation）

### R01 - 依据溯源校验

**目的**：确保诊断依据 100% 来自用户实际输入，杜绝"幻觉依据"。

**校验方法**：
1. 提取输出中每条诊断的每个依据项
2. 逐一检查该依据项是否能在用户输入的原文中找到对应信息
3. 允许合理推断（如"血糖升高"可基于"FPG 8.2 mmol/L"），但不允许无中生有

**不通过条件**：
- 依据项引用了输入中完全不存在的症状、指标、体征或检查结果
- 依据项对输入信息做了实质性歪曲（如把"轻度升高"描述为"显著升高"）

**处置**：
- 删除不合规的依据项
- 若某条诊断的全部依据项均被删除，则删除该诊断
- 记录修正日志

### R02 - 矛盾检测

**目的**：确保诊断结论不与输入中的明确信息冲突。

**校验方法**：
1. 识别输入中的明确阳性/阴性发现
2. 检查每条诊断结论是否与这些发现矛盾

**典型矛盾场景**：
- 诊断"急性阑尾炎"但输入明确记录"阑尾已切除"
- 诊断"妊娠相关疾病"但患者为男性
- 诊断"肝硬化"但肝功能指标完全正常且无影像学支持

**处置**：
- 直接矛盾 → 移除该诊断
- 部分矛盾 → 降级可能性评估等级，并标注矛盾点

### R03 - 重复检测

**目的**：避免同一疾病以不同名称多次出现，造成信息冗余。

**常见重复模式**：
- 中英文名称重复："2型糖尿病" / "T2DM" / "Type 2 Diabetes"
- 上下位关系重复："高血压" / "原发性高血压" / "高血压病3级"
- 疾病+并发症混淆："糖尿病" 和 "糖尿病肾病"（后者是并发症，不算重复）

**处置**：
- 合并为最准确的标准名称
- 保留依据最充分的条目

### R04 - 红旗遗漏检测

**目的**：确保危急征象不被遗漏。

**校验方法**：
1. 扫描输入信息中的以下红旗征象关键词/模式：
   - 急性胸痛、胸闷、呼吸困难
   - 突发头痛、意识障碍、偏瘫
   - 高热 + 血压下降 / 感染征象
   - 急性腹痛 + 腹膜刺激征
   - 危急实验室值（血钾异常、血糖极端值、肌钙蛋白升高等）
2. 若输入中存在红旗征象，检查诊断清单中是否包含对应的鉴别疾病

**处置**：
- 遗漏 → 强制在诊断清单中补充对应鉴别
- 同时确保报告开头有 🚨 急症提示

### R05 - 最低鉴别数校验

**规则**：任何输出路径，鉴别诊断数量不得少于 3 个。

**处置**：不足 3 个 → 基于输入信息补充合理的鉴别诊断。

### R06 - 缺失路径完整性

**规则**：缺失信息报告中，每条缺失项必须包含完整的依据路径。

**依据路径格式**：`症状A + 体征B → 疑似疾病C → 确诊需指标D`

**处置**：缺少依据路径 → 补充推理链路。

### R07 - 概率排序校验

**规则**：诊断列表按流行病学概率从高到低排序。罕见病不应排在常见病前面，除非有明确的强支持证据。

**校验方法**：
1. 评估每条诊断的流行病学发病率层级（常见 / 少见 / 罕见）
2. 若罕见病排在常见病前面，检查是否有≥2条强支持依据

**处置**：无充分理由的罕见病前置 → 重新排序。

### R08 - 置信度门控（信息充足性校验）

**目的**：当所有疑似诊断的可能性仅为"中"或"低"时，判定为信息不充足，禁止输出疑似诊断报告，强制转入缺失信息识别路径。

**触发场景**：无当前诊断的疑似诊断路径。

**校验方法**：
1. 检查 Phase 2 推理产出的所有疑似诊断的可能性评估
2. 取最高可能性等级

**判定规则**：

| 最高可能性 | 判定结果 | 输出路径 |
|-----------|---------|---------|
| 高 | 信息充足 | 正常输出疑似诊断报告（仅输出"高"可能性条目） |
| 中 | 信息不充足 | **禁止输出疑似诊断报告** → 强制转入缺失信息报告（可附方向提示） |
| 低 | 信息严重不充足 | **禁止输出疑似诊断报告** → 强制转入缺失信息报告（不附方向提示） |

**设计理由**：
- 低置信度的诊断预测准确度过低，输出后会误导临床决策
- 此时正确做法是明确告知"缺什么信息"以及"为什么需要"，引导医生补充关键信息后再做判断
- "中"可能性时允许附加方向提示（标注为非诊断结论），帮助医生理解推理思路

**处置**：
- 最高可能性为"中"或"低" → 将输出路径从"疑似诊断报告"切换至"缺失信息报告"
- 原本生成的疑似诊断假设列表用于反向推导缺失信息（从假设出发，分析确诊还需要哪些信息）
- 最高可能性为"中"时，在缺失信息报告末尾附加方向提示，并明确标注：
  ```
  ⚠️ 以下仅为初步方向提示，非诊断结论，需补充上述缺失信息后重新评估。
  ```

## 第二层：模型交叉验证（Model Cross-Validation）

### 触发条件

以下任一情况触发第二层：
1. 第一层 R01-R08 全部通过（常规流程）
2. 第一层规则不存在对应的判断项（如"该诊断在临床上是否合理"——无硬规则可判）
3. 第一层规则无法做出明确判断（边界情况）

### 验证维度

以"质疑者/审查者"视角重新审视推理结果：

#### V01 - 诊断-证据一致性
- 每条诊断的依据是否充分支持该诊断？
- 有无被忽视的反对证据？
- 依据的权重是否合理？（如仅凭一个非特异性症状就给"高可能性"评估）

#### V02 - 鉴别完备性
- 是否遗漏了临床上容易混淆的重要鉴别诊断？
- 鉴别诊断之间是否覆盖了不同病因机制？（感染/自身免疫/肿瘤/代谢等）

#### V03 - 一元论合理性
- 用单一疾病解释所有表现是否牵强？
- 是否存在无法被当前诊断解释的"多余"症状？
- 是否应该考虑共病/多系统疾病？

#### V04 - 严重度匹配
- 风险等级评估（高/中/低）是否与临床情况匹配？
- 是否低估了严重情况？

#### V05 - 建议可行性
- 推荐的进一步检查是否在临床上合理且可执行？
- 是否建议了过于激进或不必要的有创检查？

### 处理方式

- 发现问题 → 标注具体哪个维度（V01-V05）出了问题
- 修正对应结论
- 修正后的结果进入第三层

## 第三层：输出完整性终检（Final Gate）

### 检查清单

最终放行前，逐项确认：

| 序号 | 检查项 | 标准 |
|------|--------|------|
| F01 | 依据非空 | 每条诊断都有 ≥1 条明确依据 |
| F02 | 零臆测 | 所有依据 100% 可溯源至用户输入 |
| F03 | 红旗标注 | 红旗征象（如存在）已在报告开头 🚨 醒目标注 |
| F04 | 安全声明 | 报告末尾已附安全声明 |
| F05 | 格式合规 | 输出结构符合 `references/output_templates.md` 模板 |
| F06 | 风险标注 | 误诊/漏诊场景已标注风险等级 |
| F07 | 依据路径 | 缺失信息场景中每条缺失项有完整推理链路 |

### 处理方式

- 全部通过 → **放行输出**
- 任一项不通过 → 回退至第二层修正，修正后重新终检
- 最多回退 2 次。若第 3 次仍不通过，输出时在报告头部标注：

```
⚠️ **验证提示**：以下结论存在不确定性，建议结合临床实际审慎参考。
具体不确定项：[列出未通过的检查项]
```

## 验证流程图

```
Phase 2 产出的推理结果
        │
        ▼
  ┌───────────┐
  │ 第一层     │  逐条执行 R01-R08
  │ 规则验证   │  ─── 命中 → 自动修正
  └─────┬─────┘  ─── R08 命中 → 切换至缺失信息路径
        │        ─── 规则不存在/无法判断 → 标记进入第二层
        ▼
  ┌───────────┐
  │ 第二层     │  以质疑者视角审视 V01-V05
  │ 模型兜底   │  ─── 发现问题 → 修正
  └─────┬─────┘  ─── 无问题 → 放行至第三层
        │
        ▼
  ┌───────────┐
  │ 第三层     │  终检清单 F01-F07
  │ 完整性终检 │  ─── 全通过 → 放行输出 ✅
  └─────┬─────┘  ─── 不通过 → 回退第二层（最多2次）
        │               ─── 超限 → 带警告输出 ⚠️
        ▼
    【最终输出】
```
