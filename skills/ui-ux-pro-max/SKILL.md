# UI/UX Pro Max

用于快速给出可落地、可实现、有明确设计方向的界面方案，而不是堆大而全的设计百科。入口只保留触发条件、最小工作流和判断优先级。

## 何时使用

- 新做页面、组件、后台、落地页、移动端界面
- 用户要“更好看”“更专业”“更高级”“更一致”
- 需要检查可访问性、触达面积、层级、留白、动画
- 需要在 React / Vue / Next.js / Tailwind / shadcn 等栈里落实现代 UI

## 默认原则

优先级从高到低：

1. 可访问性
2. 交互可用性
3. 性能与响应式
4. 信息层级与布局
5. 风格、配色、排版
6. 动效和视觉强化

先解决“能不能用、看不看得清、点不点得到”，再谈风格。

## 设计方向

编码前先确定一个清晰方向，不要直接套通用模板：

- 目的：界面要解决什么问题，用户要完成什么任务
- 用户：高频操作人员、管理者、外部访客、运营分析人员等
- 语气：克制高效、专业可信、编辑感、科技感、轻量活泼、数据密集等
- 记忆点：这个页面最应该被记住的一个视觉或交互特征
- 约束：当前技术栈、设计系统、性能、可访问性、移动端需求

后台 / 运营 / SaaS 页面优先保持克制、密集、可扫描；landing page、作品集、活动页可以更大胆。不要把营销页的 hero、装饰卡片和大面积氛围图套到高频业务后台里。

## 快速检查

### 可访问性

- 正文字体对比度至少接近 4.5:1
- 图标按钮要有可理解标签
- 焦点态清晰可见
- 键盘顺序与视觉顺序一致
- 重要图片要有替代文本

### 交互

- 可点击区域尽量不小于 `44x44`
- 异步按钮有 loading 和禁用态
- 错误信息靠近问题位置
- Hover 不是唯一反馈，移动端要能触发

### 布局

- 页面有稳定的主次层级
- 控件对齐、间距、圆角、阴影保持一致
- 避免无意义的嵌套与横向滚动
- 表格、表单、卡片留白要统一

### 视觉质量

- 明确字体层级，避免全页面字号和字重混乱
- 配色要有主色、辅助色、状态色和中性色，不要只堆单一色系
- 背景、分隔、阴影、边框服务信息层级，不做无意义装饰
- 动效只用在状态变化、进入反馈、加载过渡等关键时刻
- 页面要有一个明确设计记忆点，但不要牺牲业务可读性

## 默认工作流

### 1. 判断任务类型

- 视觉改版
- 可用性修复
- 设计系统定义
- 组件级重构
- 图表 / 数据展示优化

### 2. 先给设计方向

至少明确：

- 产品类型
- 目标用户
- 主要场景
- 风格关键词
- 记忆点或差异化方向
- 技术栈

如果用户没给栈，优先按当前项目已有栈落地。

### 3. 先给一版系统化建议

默认输出至少包含：

- 风格方向
- 主色 / 辅色 / 状态色
- 字体层级
- 间距与圆角建议
- 核心组件规范
- 需要避免的反模式

可按需补读：

- 风格预设：`references/style-presets.md`
- 设计 token：`references/design-tokens.md`
- 示例模板：`references/examples.md`

### 4. 再补局部细节

按需补：

- 图表类型选择
- 表单体验优化
- 表格密度与阅读性
- 动效与微交互
- 暗色模式

## 技术落地建议

- 现有项目优先保持原栈一致，不强推新框架
- Tailwind / shadcn 项目优先用 token 和组件组合
- React / Umi 项目优先做局部重构，不一次性推翻页面
- 动画优先用 `transform` 和 `opacity`
- 需要代码时必须产出真实可运行结构，不只给视觉描述
- 现有设计系统已存在时，优先用 token、组件组合和局部样式覆盖，不另起一套视觉体系

## 输出期望

默认回答尽量包含：

- 问题诊断
- 优先级最高的 3 到 5 项改进
- 可直接落地的组件 / 布局建议
- 如需代码，给出与当前栈一致的实现

## 不要这样做

- 只讲风格，不讲使用成本
- 为了好看牺牲可读性
- 大量使用低对比度文本
- 给出和当前技术栈不匹配的方案
- 在没有业务理由时过度动画化
- 使用泛化 AI 风格：紫蓝渐变、玻璃拟态堆叠、无业务含义的大卡片、千篇一律的 hero
- 不看产品类型就套同一套视觉模板
- 只做静态好看，不处理 loading、empty、error、disabled、hover、focus 等状态

## 适合继续深挖的方向

- 组件设计系统
- 页面级重构
- dashboard 信息架构
- landing page 转化优化
- 无障碍审计
